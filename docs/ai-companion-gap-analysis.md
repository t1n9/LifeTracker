# AI 陪伴系统差距分析

## 一、总体评估

当前 AI 系统处于"勉强可用"阶段——能对话、能调用工具、能主动发消息，但离 PRD 描述的"全天陪伴"差得很远。核心问题：**没有状态记忆、没有情境感知、没有多步对话流**。

---

## 二、PRD 功能逐项检查

| PRD 功能 | 实现状态 | 说明 |
|---|---|---|
| 晨间流 - 起床时间引导 | ⚠️ 半实现 | 只发了单条消息，LLM 可能在消息里问"几点起的"，但后端没有追踪对话状态，无法多轮推进 |
| 晨间流 - 任务规划 | ❌ 未实现 | PRD 要求根据学习计划注入今日任务槽，当前后端只查了 goal 和 pendingTask 计数，不做任何自动化 |
| 晨间流 - 开启番茄引导 | ❌ 未实现 | 没有任务规划完成后的下一步引导 |
| 番茄结束推送 | ⚠️ 半实现 | 能发消息，但消息是 LLM 自由生成的简单文本，没有根据番茄数量、任务进度变化策略 |
| 番茄结束冷却 | ✅ 已实现 | 同任务间隔 >= 2 个番茄才重复推送 |
| 任务完成推送 | ❌ 未实现 | `task_done` 事件类型定义了但从未 dispatch，PRD 要求的"继续还是休息"、"全清了要复盘吗"等消息不存在 |
| 晚间复盘流 | ❌ 未实现 | Phase 2，但连事件 dispatch 都没有 |
| 动画提示 | ✅ 已实现 | 呼吸灯 + 闪烁 |
| 删除 DayReflection 组件 | ✅ 已实现 | |

**Phase 1 完成度：约 30%**

## 三、已知缺陷清单

### 🔴 P0：重复晨间问候

**现象**：用户从别的页面回到首页，每次 Dashboard 重新挂载都会重新 dispatch `morning` 事件，导致 AI 重复发"早上好"。

**根因**：`Dashboard.tsx` 第 127 行 `useEffect([], [])` 只在组件挂载时检查 `dayStart`，没有记录"今天已发送过问候"。Next.js 客户端路由每次导航都卸载/重建组件。

**修复方向**：在 localStorage 或 sessionStorage 记录 `proactive_morning_sent_date`，当天已发送则跳过。

### 🔴 P0：主动消息没有情境上下文

**现象**：AI 发的消息像模板填充，不知道用户今天做了什么。

**根因**：`buildProactiveContext` 只拉取了最基础的数据（goal名称、待办任务数），没有包含：
- 用户今天的实际任务列表（标题 + 完成状态）
- 今天的番茄记录
- 今天的学习时长
- 用户的长期记忆（profile/summary）
- 最近的对话历史
- 时间上下文（现在几点了，用户今天用了多久）

**结果**：LLM 只能编一句通用的话，做不到"你今天已经完成了 3 个任务，很高效"这种真实鼓励。

**修复方向**：`buildProactiveContext` 应该调用 `buildProfileContext` + `buildShortTermContext` + 完整的今日概况（已完成/未完成任务、番茄数、学习时长）。

### 🟡 P1：没有多步对话流

**现象**：晨间流每次都是"发一条消息完事"，不会根据用户回复推进。

**根因**：后端没有对话状态机。`handleProactive` 生成消息后写入 `toolCalls: { proactive: true }` 标记为不进入后续上下文，这个标记本意是防止 LLM 把旧的主动消息当指令——但同时也让 AI 完全忘记了上一轮在聊什么。

PRD 设计的晨间流是：
```
AI: 早上好，几点起的？
用户: 7点半
AI: 今天有学习计划吗？ → start_day → 查 study plan → 注入任务
```

实际是：
```
AI: 早上好，几点起的？
用户: 7点半
AI: ← 不知道刚才说了什么，新一轮对话完全没有上下文
```

**修复方向**：需要一个轻量级的对话状态追踪（session 内 `Map<userId, { phase, data }>`），不用持久化，但至少在一个浏览器会话内能记住晨间流走到了第几步。

### 🟡 P1：任务完成事件未 dispatch

**现象**：用户完成一个任务，AI 毫无反应。

**根因**：`task_done` 事件类型在 `agent-events.ts` 里定义了，但全代码库没有任何地方 dispatch 它。`PendingTasks.tsx` 完成任务后没有调用 `PROACTIVE_TRIGGER_EVENT`。

**修复方向**：在 `PendingTasks.tsx` 的 `handleComplete` 中 dispatch `task_done` 事件。

### 🟡 P1：主动消息不使用用户记忆

**现象**：用户设置了长期目标（如"备考福建事业单位"），但 AI 从不主动提，也不会根据目标给鼓励。

**根因**：`buildProactiveContext` 没有调用 `buildProfileContext`。用户的 `userGoals`、`userSettings`、`agentProfile` 全都没有注入到主动消息的上下文中。

### 🟢 P2：提示词过于通用

后端 `buildProactiveSystemPrompt` 的 prompt 是所有 trigger 共享的：
```
你是 LifeTracker 的专属陪伴助手。你现在需要主动向用户发送一条消息。
```

没有根据 trigger 类型、时间、用户状态做区分。不同场景应该有不同的人格和话术：
- 晨间：温暖、有活力、"新的一天开始了"
- 番茄结束：简短、干脆、"干得不错"
- 晚间：平静、反思、"今天辛苦了"

### 🟢 P2：番茄推送消息没有差异化策略

PRD 设计了根据番茄数量的差异化消息：
- 第 1 个："专注了 25 分钟，进展怎么样？"
- 第 3 个："连续 3 个番茄了 💪 要不要起来动一动？"
- 接近完成时："感觉快收尾了？完成了告诉我"

实际实现：把 `pomodoroCount` 传给 LLM，让 LLM 自由发挥。结果是每次消息风格不一致，有时太长，有时太短，没有可控性。

### 🟢 P2：没有运动记录感知

PRD 要求番茄推送时如果用户今天没运动，提醒动一动。但 `buildProactiveContext` 里运动记录是作为 context 传给了 LLM，LLM 不一定能在生成时正确使用这个信息。

### 🟢 P2：意图识别纯正则匹配

`agent-intent.utils.ts` 用正则匹配用户意图（如 `TASK_COMPLETION_PATTERNS`）。对于简单指令可以，但用户说"那个资料分析的活我搞完了"这种自然语言，正则完全匹配不到，只能靠 LLM 的 function calling。但 LLM 的判断有时不稳定。

## 四、架构层面的差距

| 维度 | 当前 | 理想 |
|---|---|---|
| 对话状态 | 无状态，每次对话独立 | 会话内有状态的阶段追踪（morning flow 第几步） |
| 情境感知 | 无，仅查 DB | 完整的用户画像 + 今日实时状态 + 对话上下文 |
| 主动推送 | 单次 LLM 调用生成 | 有策略的模板 + LLM 润色（保证质量可控） |
| 记忆系统 | 存在但未用于主动推送 | 主动推送也读取用户偏好和目标 |
| 事件驱动 | 只有 morning 和 pomodoro_done | 完整的事件网：任务完成、学习进度、时间节点 |
| 冷却控制 | 前端 session 内 Map | 后端数据库记录，跨设备一致 |

---

## 五、建议的修复优先级

### 立即修复（今天可做）
1. **修复重复晨间问候** — localStorage 记录发送日期
2. **dispatch task_done 事件** — PendingTasks 完成时触发

### 短期（1-2 天）
3. **加强 proactive context** — 注入 profile、今日任务列表、已完成统计
4. **差异化番茄消息策略** — 不依赖 LLM 自由发挥，用模板 + LLM 润色
5. **对话状态机**（会话级，不持久化）— 让晨间流能多步推进

### 中期（Phase 2）
6. 晚间复盘流
7. 后端冷却控制（持久化到 DB）
8. 基于学习进度的智能推送（"今天的任务完成 80% 了"、"这个科目今天还没碰"）
